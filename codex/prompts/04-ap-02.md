Implement AP-02 only after AP-01B has been reviewed and merged.

## Objective

Prove that ServiceFabric can publish selected capabilities from:

```text
a ServiceFabric-hosted application
an independently operated HTTP application
an independent LangGraph project
```

and make them usable through:

```text
the installed ServiceFabric CLI
an independent LangGraph consumer
```

Build on the existing gateway. Do not create a second tool-running system.

## Branch

Create from updated `main`:

```text
feature/ap-02-application-capability-connections
```

## Capability providers

### Hosted provider

Use the Data Analysis Workspace capability:

```text
data.profile_dataset
```

### Independent service

Add:

```text
examples/independent-market-service/
```

It remains independently operated and provides a reviewed read-only action:

```text
market.get_price_snapshot
```

Use deterministic local demonstration data. Do not depend on an internet market-data provider for acceptance tests.

### Independent LangGraph project

Add:

```text
examples/independent-analysis-graph/
```

It loads a selected ServiceFabric toolset and combines:

```text
data.inspect_dataset
data.profile_dataset
data.create_report
```

Publish the complete workflow back as:

```text
analysis.prepare_dataset_report
```

The graph must call ServiceFabric through a supported client connection. It must not import underlying application implementations.

## Toolsets

Create real toolsets:

```text
data-analysis
market-read-only
analysis-workflows
```

Toolsets must contain selected capabilities, not complete application access.

## Connection profiles

Add user-manageable local connection profiles for:

```text
installed CLI
independent Python or LangGraph project
```

A profile records:

```text
consumer name
consumer type
assigned toolsets
visible capabilities
local connection method
status
last activity
```

## Local connection method

Implement the smallest supported local transport necessary for an independent project.

Use a loopback-only connection.

It must:

```text
authenticate or identify a configured local consumer
list its assigned capabilities
call through the existing gateway
follow operations
request cancellation
return safe errors
```

Do not bind publicly by default.

Do not implement broad Codex, Claude Code or Gemini configuration in this milestone.

Do not add a second implementation of MCP. Reuse the V4 gateway where MCP projection is required.

## CLI journey

The user can run:

```bash
servicefabric connections create local-cli \
  --type cli \
  --toolset data-analysis

servicefabric connections inspect local-cli

servicefabric --connection local-cli tools list
servicefabric --connection local-cli call data.inspect_dataset \
  --input '{"dataset":"transactions.csv"}'
```

## Independent LangGraph journey

The example project uses a simple supported interface such as:

```python
tools = load_servicefabric_toolset(
    connection="analysis-graph",
    toolset="data-analysis",
)
```

It must work while running as a separate project/process.

## Application startup integration

When a selected capability belongs to a stopped hosted application:

1. identify its source application;
2. evaluate the application’s startup policy;
3. display the expected resource effect where confirmation is required;
4. start the application;
5. wait for health;
6. call the capability.

A failed startup must return a clear capability-unavailable result.

## External application review


Do not import every endpoint automatically.

## End-to-end proof

The user must be able to:

1. call a hosted application capability from the CLI;
2. call an independent service capability from the CLI;
3. load the hosted capability in an independent LangGraph project;
4. follow its longer operation;
5. publish the graph as a composite capability;
6. call the composite capability from the CLI;
7. identify the source application and consuming connection;
8. verify that unselected external actions remain unavailable.
