## Demonstration application

Add:

```text
examples/data-analysis-workspace/
```

Modules:

```text
standard React or Svelte interface
FastAPI API
persistent dataset storage
Python analysis worker
bounded local work dispatcher
report renderer
capability adapter
```

Publish:

```text
data.list_datasets
data.inspect_dataset
data.profile_dataset
data.create_report
```

`data.profile_dataset` must be a genuine longer-running local operation that reports progress and supports cancellation.

Do not use artificial sleep as the main business operation. Use a bounded deterministic dataset-analysis task.

## User journey

The following must work:

```bash
servicefabric apps create data-workspace \
  --template data-analysis-workspace

servicefabric apps build data-workspace
servicefabric apps plan-start data-workspace
servicefabric apps start data-workspace
servicefabric apps resources data-workspace
```

Then:

```bash
servicefabric call data.profile_dataset \
  --input '{"dataset":"transactions.csv"}'
```

The call returns an operation reference.

The user can run:

```bash
servicefabric operations get OPERATION_ID
servicefabric operations watch OPERATION_ID
servicefabric operations cancel OPERATION_ID \
  --reason "Manual cancellation test"
```

Persistent uploaded or seeded data must remain available after application restart.

## Hosting policies

Add only the local policies exercised here:

```text
manual start
start on first capability call
stop after inactivity
remain running
```

Do not build a distributed scheduler.

## Operation bridge

Application work must map to existing ServiceFabric operations.

ServiceFabric must not edit application job data directly.

The application adapter reports:

```text
accepted
running
progress
completed
failed
cancellation requested
cancelled
```

Cancellation must reach the worker rather than only changing a ServiceFabric status label.
