# Service Fabric: Real-Time Architecture & Integration

This document outlines how to leverage the **Service Fabric Central Gateway** (FastAPI) for real-time WebSockets and instant messaging across all shards.

## 1. Architectural Overview
The Service Fabric uses a hub-and-spoke model for real-time events:
- **Central Hub (FastAPI)**: Manages all persistent WebSocket connections (`ws://.../api/v1/ws/events/`).
- **Shards (Flask/Django/React)**: Act as "producers" or "consumers" of events.
- **Protocols**: Shards are encouraged to move away from standalone Socket.IO servers in favor of this centralized raw-WebSocket hub.

## 2. Using the FastAPI Utility in Flask Shards

### Backend (Python): Pushing Events
Every Flask shard following the `flask_base` template has access to the `FabricSDK` via the `fabric` object.
```python
from _shared.utils.fabric_sdk import fabric

@bp.route('/api/update')
def update():
    # ... logic ...
    # Broadcast an event to all connected clients in the Fabric
    fabric.broadcast("my_app_updated", {"id": 123})
    return jsonify({"status": "ok"})
```

### Frontend (JavaScript): Receiving Events
Shards should use the `FabricClient` to connect to the Gateway hub.

- **Vanilla JS/HTML Shards**: Use `_shared/utils/fabric-client.js` via a script tag.
- **React/Svelte/Vite Shards**: Use `_shared/utils/fabric-client.ts` via ESM import.

**In your `index.html` (Vanilla):**
```html
<script src="/app/core/_shared/utils/fabric-client.js"></script>
```

**In your React/Svelte Component:**
```javascript
import { fabric } from '../../_shared/utils/fabric-client.ts';
```

## 3. Integrating Messaging (Refactoring)
The `6_service_catalog/messaging` shard currently uses a standalone `flask-socketio` implementation. To integrate with the central Fabric Gateway:

1. **Remove `flask-socketio`**: Delete the standalone socket server.
2. **Standardize Broadcasts**: In `routes.py`, call `fabric.broadcast("new_message", {...})` after saving a message to the database.
3. **Connect to Hub**: In `chatroom.html`, replace the `socket.io` client with `fabric-client.js`.
4. **Subscribe to "new_message"**: Use `window.fabric.onEvent` to listen for the incoming message event from the Gateway.

## 5. Troubleshooting & common Pitfalls
- **TemplateNotFound Errors**: If a shard fails to load its templates, ensure `5_core_services/flask_base/app/__init__.py` has a `ChoiceLoader` that includes the `original_loader`. Without this, Flask ignores blueprint-specific template folders.
- **Database Protocol**: While SQLAlchemy 1.4+ technically requires `postgresql://`, ensure the platform bootstrap logic supports `postgres://` for backward compatibility with older `.env` files to avoid skipping service registration.
- **Dynamic Mounting**: Services are only mounted if `DATABASE_URL` is detected. Check logs for "Caricamento Servizi Dinamici saltato" if your shard isn't appearing.

## 6. Fabric Manifest Mandate
Every application in the `6_service_catalog/` (shards) must contain a `fabric-manifest.json` file.

4.  **Preserve the manifest** during refactoring or updates.
